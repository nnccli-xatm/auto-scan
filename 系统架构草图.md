## 二、运行时架构

### 2.1 进程与协程模型

```
主进程 (Main Process)
│
├─ 主协程 (Main Goroutine)
│  └─ 信号处理 (SIGINT/SIGTERM)
│  └─ 配置热重载
│
├─ API服务器协程池 (Gin)
│  ├─ HTTP请求处理 (多协程)
│  └─ WebSocket连接管理
│
├─ 设备发现协程
│  └─ mDNS监听 (Bonjour/Zeroconf)
│  └─ WS-Discovery监听 (WSD)
│
├─ 设备监控协程 (每个设备一个)
│  ├─ HP 750 [192.168.3.11] │ 轮询状态 │ ADF检测
│  ├─ Canon [192.168.3.12]  │ 轮询状态 │ ADF检测
│  └─ ...
│
├─ 任务调度协程
│  └─ 任务队列管理
│  └─ 扫描执行器 (限制并发数)
│
├─ 存储管理协程
│  ├─ 文件写入队列
│  ├─ 校验和计算 (后台)
│  └─ 备份同步 (后台)
│
├─ 审计日志协程
│  └─ 批量写入SQLite
│  └─ 日志轮转
│
└─ 系统监控协程
   ├─ 性能指标采集
   └─ 资源使用监控

协程总数预估:
- 基础: 10-15个 (API/发现/调度/存储/日志)
- 每设备: 1-2个 (监控+可能的扫描)
- 50设备总协程: 110-115个
```

### 2.2 数据流

```
扫描触发流程:
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  设备ADF    │     │  扫描任务   │     │  文件存储   │
│  状态变化   │────▶│  调度执行   │────▶│  与元数据   │
└─────────────┘     └─────────────┘     └─────────────┘
       │                   │                   │
       ▼                   ▼                   ▼
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  eSCL请求   │     │  HTTP下载   │     │  完整性     │
│  创建任务   │     │  图像数据   │     │  校验       │
└─────────────┘     └─────────────┘     └─────────────┘
       │                   │                   │
       ▼                   ▼                   ▼
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  返回任务ID │     │  PDF/JPG    │     │  审计日志   │
│             │     │  格式处理   │     │  记录       │
└─────────────┘     └─────────────┘     └─────────────┘

文件处理流水线:
原始数据 → 格式识别 → PDF拆分(如需要) → JPG转换 → 校验和计算 → 双副本写入 → 元数据记录
   │           │              │                │            │            │            │
   ▼           ▼              ▼                ▼            ▼            ▼            ▼
流式下载   Content-Type    pdf2image       imaging      Blake3      本地+NAS     SQLite

## 三、模块详细设计

### 3.1 设备管理模块

```
DeviceManager
│
├─ DiscoveryService (发现服务)
│  ├─ mDNSDiscovery
│  │  ├─ 监听 _scanner._tcp
│  │  ├─ 监听 _uscan._tcp
│  │  └─ 监听 _ipp._tcp
│  └─ WSDDiscovery
│     ├─ WS-Discovery Probe
│     └─ SOAP解析
│
├─ DeviceRegistry (设备注册表)
│  ├─ 内存缓存 (ristretto)
│  └─ 持久化存储 (SQLite)
│
├─ DeviceMonitor (设备监控)
│  ├─ 状态轮询器 (每2秒)
│  ├─ ADF检测器
│  └─ 故障检测器
│
└─ ProtocolFactory (协议工厂)
   ├─ eSCLDriver
   ├─ WSDDriver
   └─ SANEDriver (Linux)
```

### 3.2 任务调度模块

```
ScanScheduler
│
├─ TaskQueue (任务队列)
│  ├─ 优先级队列 (设备优先级)
│  ├─ 去重机制 (同设备防并发)
│  └─ 超时处理 (任务TTL)
│
├─ ExecutorPool (执行池)
│  ├─ 最大并发数: 5 (可配置)
│  ├─ 协程池管理
│  └─ 优雅关闭
│
├─ ScanExecutor (扫描执行器)
│  ├─ 任务创建 (eSCL POST)
│  ├─ 图像下载 (流式)
│  ├─ 格式处理 (PDF→JPG)
│  └─ 异常处理 (重试/失败)
│
└─ Pipeline (处理流水线)
   ├─ Downloader (下载器)
   ├─ FormatConverter (格式转换)
   ├─ QualityChecker (质量检查)
   └─ StorageWriter (存储写入)
```

### 3.3 存储管理模块

```
StorageManager
│
├─ MetadataStore (元数据存储)
│  └─ SQLiteManager
│     ├─ 连接池
│     ├─ 事务管理
│     └─ 查询优化
│
├─ FileStore (文件存储)
│  ├─ LocalFS (本地文件系统)
│  │  ├─ 目录结构管理
│  │  ├─ 原子写入 (临时文件+重命名)
│  │  └─ 空间监控
│  ├─ NASStore (网络存储)
│  │  ├─ SMB/CIFS客户端
│  │  └─ 异步同步
│  └─ BackupManager (备份管理)
│     ├─ 双副本策略
│     ├─ 增量备份
│     └─ 定期校验
│
├─ IntegrityChecker (完整性检查)
│  ├─ Blake3校验和计算
│  ├─ 定期扫描验证
│  └─ 损坏文件恢复
│
└─ QuarantineManager (隔离管理)
   ├─ 可疑文件检测
   ├─ 隔离目录管理
   └─ 自动清理策略
```

### 3.4 审计日志模块

```
AuditLogger
│
├─ LogCollector (日志收集)
│  ├─ 异步缓冲 (channel)
│  ├─ 批量写入 (每100条或5秒)
│  └─ 级别过滤 (Info/Warning/Error)
│
├─ LogStorage (日志存储)
│  ├─ SQLite表 (audit_logs)
│  ├─ 分区策略 (按月分区)
│  └─ 索引优化 (timestamp, event_type)
│
├─ LogRetention (日志保留)
│  ├─ 热数据: 30天 (SQLite)
│  ├─ 温数据: 90天 (压缩归档)
│  └─ 冷数据: 7年 (离线存储)
│
└─ LogQuery (日志查询)
   ├─ 时间范围查询
   ├─ 设备过滤
   ├─ 事件类型过滤
   └─ 导出功能 (CSV/PDF)
```

```

   │

   └──▶ macOS构建 (GOOS=darwin GOARCH=amd64/arm64)

## 七、技术选型总结

| 层级 | 组件 | 技术 | 理由 |
|------|------|------|------|
| **语言** | 主要语言 | Go 1.22+ | 静态编译，高性能，跨平台友好 |
| **网络** | HTTP框架 | Gin | 轻量，高性能，中间件丰富 |
| | mDNS发现 | zeroconf | 标准Bonjour实现 |
| **存储** | 元数据 | SQLite | 嵌入式，零配置，事务支持 |
| | 文件系统 | 本地+NAS | 简单可靠，易于备份 |
| | 缓存 | ristretto | 高性能内存缓存 |
| **图像** | 处理 | imaging | 纯Go，无CGO依赖 |
| | PDF处理 | pdfcpu | 纯Go，轻量 |
| | 转换 | pdf2image桥接 | 处理复杂PDF |
| **日志** | 应用日志 | Zap | 结构化，高性能 |
| | 审计日志 | SQLite | 结构化查询，易于分析 |
| **监控** | 指标 | Prometheus | 行业标准 |
| | 告警 | Alertmanager | 灵活告警策略 |
| **构建** | 构建工具 | Mage | Go语言编写构建脚本 |
| | CI/CD | GitHub Actions | 集成GitHub，免费 |
| **部署** | Windows | Windows Service | 系统集成 |
| | Linux | systemd | 标准服务管理 |

---

**架构设计原则**:
1. **简单优先**: 避免过度设计，单体架构起步
2. **可观测**: 全面日志、指标、追踪
3. **容错性**: 单点故障不影响整体
4. **可演进**: 模块化设计，未来可拆分
5. **安全内建**: 安全不是附加功能

**下一步**: 基于本架构进行详细设计，包括API设计、数据库Schema、界面原型。
