# 扫描设备自动任务系统 - 技术栈详细对比

## 一、编程语言深度对比

### 1.1 Go vs Python 详细对比

| 对比维度 | Go | Python | 胜出 |
|---------|-----|--------|------|
| **开发效率** | 中等（需编译） | 高（解释执行） | Python |
| **运行时性能** | 高（接近C） | 中等（CPython） | Go |
| **内存占用** | 低（20-50MB） | 中（50-100MB） | Go |
| **跨平台编译** | 简单（GOOS/GOARCH） | 复杂（PyInstaller） | Go |
| **部署友好性** | 单文件，无依赖 | 需Python环境或打包 | Go |
| **扫描仪生态** | 需自行封装 | SANE成熟，eSCL库多 | Python |
| **图像处理** | 库较少（imaging） | Pillow/pdf2image成熟 | Python |
| **类型安全** | 静态类型，编译检查 | 动态类型，运行错误 | Go |
| **并发模型** | 协程（goroutine） | asyncio/multiprocessing | Go |
| **安全软件友好** | 静态编译，低误报 | 解释器行为易被误报 | Go |
| **长期维护** | 版本稳定，向后兼容 | 2/3分裂，依赖地狱 | Go |

**综合评分**: Go 8:3 Python → **推荐Go**

## 二、扫描协议技术细节

### 2.1 eSCL协议详解

**全称**: Extensible Scanner Control Language  
**基础**: IPP (Internet Printing Protocol) + 扫描扩展  
**传输**: HTTP/HTTPS (端口80/443/631/8080)  
**发现**: mDNS/Bonjour (`_uscan._tcp`, `_scanner._tcp`)

**关键端点**:
```
GET  /eSCL/ScannerCapabilities   # 获取能力
GET  /eSCL/ScannerStatus         # 状态查询
POST /eSCL/ScanJobs              # 创建任务
GET  /eSCL/ScanJobs/{id}         # 任务状态
GET  /eSCL/ScanJobs/{id}/NextDocument  # 获取图像
DELETE /eSCL/ScanJobs/{id}       # 取消任务
```

**状态机**:
```
Idle → Processing → Completed/Aborted/Canceled
```

**HP实现特性**:
- 任务创建后立即返回201
- 不支持任务状态查询（405错误）
- 需直接尝试下载NextDocument
- 多页扫描返回PDF，需拆分

### 2.2 WSD协议详解

**全称**: Web Services for Devices  
**基础**: WS-Discovery + SOAP  
**发现**: UDP多播 (239.255.255.250:3702)  
**传输**: HTTP POST

**复杂度**: 高于eSCL  
**Windows**: 原生支持  
**Linux**: 需实现WS-Discovery

### 2.3 协议选择策略

```
设备发现:
  ├─ mDNS广播 → eSCL设备
  ├─ WS-Discovery → WSD设备  
  └─ 手动配置 → 尝试所有协议

设备连接:
  ├─ 优先尝试eSCL（简单可靠）
  ├─ 失败尝试WSD
  └─ Linux尝试SANE
```

## 三、存储方案详细设计

### 3.1 文件存储策略对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **本地文件系统** | 简单，速度快 | 单机，需备份 | 小规模部署 |
| **网络存储(NAS)** | 集中存储，RAID | 依赖网络 | 中规模部署 |
| **对象存储** | 高可用，扩展性 | 成本，延迟 | 大规模/云部署 |
| **分布式存储** | 企业级 | 复杂，运维高 | 超大规模 |

**推荐**: 本地 + NAS双副本（初期）

### 3.2 数据库选型对比

| 特性 | SQLite | BoltDB | PostgreSQL |
|------|--------|--------|-----------|
| **部署** | 嵌入式 | 嵌入式 | 独立服务 |
| **SQL支持** | 完整 | 无（KV） | 完整 |
| **并发写入** | 表级锁 | 串行 | 行级锁 |
| **事务** | ACID | ACID | ACID |
| **备份** | 文件复制 | 文件复制 | pg_dump |
| **资源占用** | 极低 | 极低 | 高 |
| **查询能力** | 强 | 弱（需遍历） | 强 |
| **适用场景** | 嵌入式设备 | KV为主 | 企业级 |

**推荐SQLite**: 查询需求强，单文件易备份

### 3.3 SQLite Schema设计

```sql
-- 设备表
CREATE TABLE devices (
    id TEXT PRIMARY KEY,           -- UUID
    name TEXT NOT NULL,            -- 设备名称
    ip_address TEXT NOT NULL,      -- IP地址
    protocol TEXT NOT NULL,        -- ecl/wsd/sane
    model TEXT,                    -- 型号
    status TEXT DEFAULT 'unknown', -- online/offline/busy
    last_seen DATETIME,            -- 最后在线时间
    config JSON,                   -- 配置参数
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 扫描批次表
CREATE TABLE scan_batches (
    id TEXT PRIMARY KEY,           -- UUID
    device_id TEXT REFERENCES devices(id),
    start_time DATETIME NOT NULL,
    end_time DATETIME,
    page_count INTEGER DEFAULT 0,
    status TEXT,                   -- running/completed/failed
    storage_path TEXT,             -- 存储路径
    checksum TEXT,                 -- 批次校验和
    metadata JSON                  -- 元数据
);

-- 扫描文件表
CREATE TABLE scan_files (
    id TEXT PRIMARY KEY,
    batch_id TEXT REFERENCES scan_batches(id),
    filename TEXT NOT NULL,
    page_number INTEGER,
    size_bytes INTEGER,
    checksum TEXT,                 -- SHA256
    created_at DATETIME,
    status TEXT DEFAULT 'ok'       -- ok/quarantine/deleted
);

-- 审计日志表
CREATE TABLE audit_logs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp DATETIME DEFAULT CURRENT_TIMESTAMP,
    level TEXT,                    -- info/warning/error
    event_type TEXT NOT NULL,      -- 事件类型
    device_id TEXT,
    user_id TEXT,
    message TEXT,
    details JSON,                  -- 详细信息
    ip_address TEXT                -- 客户端IP
);

-- 索引
CREATE INDEX idx_devices_status ON devices(status);
CREATE INDEX idx_batches_time ON scan_batches(start_time);
CREATE INDEX idx_files_batch ON scan_files(batch_id);
CREATE INDEX idx_audit_time ON audit_logs(timestamp);
```

### 4.2 关键接口定义

// DeviceManager 设备管理器接口
type DeviceManager interface {
    Discover(ctx context.Context) error
    Add(device DeviceInfo) error
    Remove(deviceID string) error
    Get(deviceID string) (Device, error)
    List() []Device
    StartMonitoring() error
    StopMonitoring() error
}

// Scanner 扫描执行接口
type Scanner interface {
    Scan(ctx context.Context, settings ScanSettings) (ScanResult, error)
    GetStatus() (DeviceStatus, error)
    Cancel(jobID string) error
}

// Storage 存储接口
type Storage interface {
    Save(batch ScanBatch, files []ScanFile) error
    Get(batchID string) (ScanBatch, error)
    List(filter Filter) ([]ScanBatch, error)
    Delete(batchID string) error
    Verify(batchID string) (bool, error)
}

// AuditLogger 审计日志接口
type AuditLogger interface {
    Log(event AuditEvent) error
    Query(filter LogFilter) ([]AuditEvent, error)
    Export(start, end time.Time) (io.Reader, error)
}

## 五、部署方案对比

### 5.1 Windows部署

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **Windows Service** | 系统级，开机启动 | 需管理员权限 | ⭐⭐⭐⭐⭐ |
| **计划任务** | 简单，用户级 | 非服务形态 | ⭐⭐⭐ |
| **系统托盘** | 用户交互好 | 需登录 | ⭐⭐⭐⭐ |
| **Docker Desktop** | 隔离性好 | 资源占用高 | ⭐⭐ |

**推荐**: Windows Service + 可选托盘图标

### 5.2 Linux部署

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **systemd** | 标准，强大 | 需root配置 | ⭐⭐⭐⭐⭐ |
| **Docker** | 隔离，可移植 | 额外依赖 | ⭐⭐⭐⭐ |
| **supervisord** | 简单，用户级 | 非标准 | ⭐⭐⭐ |
| **cron** | 无守护 | 不适合持续运行 | ⭐ |

**推荐**: systemd（生产）或 Docker（开发/测试）

## 六、性能基准预估

### 6.1 资源占用估算

| 场景 | 内存 | CPU | 网络 | 存储IO |
|------|------|-----|------|--------|
| **空闲** | 20-30MB | <1% | 少量心跳 | 无 |
| **单设备扫描** | 50-80MB | 10-20% | 5-10MB/s | 顺序写 |
| **10设备并发** | 100-150MB | 30-50% | 50-100MB/s | 高 |
| **峰值处理** | 200MB+ | 80%+ | 饱和 | 极高 |

### 6.2 性能优化策略

**内存优化**:
- 对象池重用（HTTP client，buffer）
- 流式处理（边下载边存储）
- 分页加载（设备列表）

**CPU优化**:
- 协程池限制并发扫描数
- 后台任务nice优先级
- 图像处理限制并发

**网络优化**:
- HTTP keep-alive
- 连接池复用
- 状态缓存（TTL 5s）
