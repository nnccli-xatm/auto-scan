### 1.2 明确排除（延后到Phase 2+）

| 功能 | 延后原因 | 计划阶段 |
|------|---------|---------|
| 富士通ScanSnap USB设备 | 需TWAIN+USB开发，复杂度高 | Phase 2 |
| TWAIN协议支持 | 仅富士通需要，暂不投入 | Phase 2 |
| ISIS协议支持 | 高端富士通fi系列，用量少 | Phase 3 |
| 高级图像处理（OCR等） | 非核心需求 | Phase 2 |
| 多节点分布式部署 | 当前单机足够 | Phase 3 |
| 移动端App | Web界面足够 | Phase 2 |

## 二、技术架构确认

### 2.1 简化后的技术栈

```
语言: Go 1.22+
主要协议: eSCL (AirPrint Scan)
辅助协议: WSD (部分理光设备)
数据库: SQLite 3.40+
Web框架: Gin
前端: Vue3 + Element Plus (可选，可先REST API)
构建: Go Modules + GitHub Actions
部署: Windows Service / Linux systemd
```

### 2.2 移除的复杂度

| 原考虑 | 当前决策 | 节省工作量 |
|--------|---------|-----------|
| TWAIN协议层 | ❌ 移除 | -2周 |
| USB设备监控 | ❌ 移除 | -1周 |
| libusb依赖 | ❌ 移除 | -3天 |
| Windows驱动管理 | ❌ 简化 | -1周 |
| macOS ImageCapture | ❌ 延后 | -1周 |

**总计节省**: 约5-6周工作量

## 四、关键技术决策确认

### 4.1 协议策略

```
设备发现:
  └─ mDNS (_uscan._tcp, _scanner._tcp)
       ├─ 自动发现HP/佳能/理光
       └─ 轮询WS-Discovery作为备选（理光部分型号）

设备连接:
  └─ 优先eSCL (HTTP 80/8080/631)
       ├─ HP: 已验证，标准实现
       ├─ 佳能: 标准eSCL
       └─ 理光: eSCL + 可选WSD
```

### 4.2 存储策略

```
存储层级:
  热存储: 本地SSD (当日扫描，快速访问)
  备份存储: NAS挂载 (自动复制，防丢失)
  
目录结构:
  ~/scan-data/
    ├── devices/
    │   └── <device-id>/
    │       └── 2026/
    │           └── 06-11/
    │               └── 14.30.25/
    │                   ├── page_001.jpg
    │                   └── metadata.json
    └── archive/ (可选，旧文件迁移)
```

### 4.3 部署形态

**Windows**:
- 单MSI安装包
- 包含: 主程序 + Web前端 + 默认配置
- 安装为Windows Service，开机自启
- 系统托盘图标（可选）

**Linux**:
- deb/rpm包
- 依赖: sqlite3, pdf2image (pip)
- systemd服务管理
- 配置文件: /etc/auto-scan/config.yaml

## 五、风险控制

### 5.1 技术风险

| 风险 | 可能性 | 缓解措施 |
|------|--------|---------|
| 理光eSCL兼容性差 | 中 | Week 1-2预留时间测试，不兼容则转WSD |
| 佳能部分型号eSCL差异 | 中 | 收集具体型号测试，维护兼容性列表 |
| 大文件PDF处理内存溢出 | 中 | 流式处理，限制并发转换数 |
| NAS备份网络故障 | 中 | 异步备份，本地先存，失败重试 |

### 5.2 进度风险

| 风险 | 应对 |
|------|------|
| Week 3设备发现延迟 | 先支持手动添加，自动发现可延后 |
| Week 5PDF转换问题 | 先用临时方案（保存PDF不转换），后续优化 |
| Week 7前端进度慢 | 先用简单HTML+JS，Vue可延后 |
