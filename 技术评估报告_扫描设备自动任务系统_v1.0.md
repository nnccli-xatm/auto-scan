## 二、技术评估维度

### 2.1 架构模式评估

#### 方案A: 单体守护进程 + 轻量级Web界面
```
┌─────────────────────────────────────┐
│         单体守护进程 (Agent)          │
│  ┌──────────┐ ┌──────────┐          │
│  │ 设备管理  │ │ 任务调度  │          │
│  └──────────┘ └──────────┘          │
│  ┌──────────┐ ┌──────────┐          │
│  │ 存储管理  │ │ Web API  │          │
│  └──────────┘ └──────────┘          │
└─────────────────────────────────────┘
```

**优点**: 部署简单，资源占用低，进程间通信开销小  
**缺点**: 功能耦合，升级需重启，横向扩展困难  
**适用**: 中小型部署（<20台设备）

#### 方案B: 分层架构（控制平面 + 数据平面）
```
┌─────────────────────────────────────┐
│      控制平面 (Control Plane)        │
│   策略管理 │ 任务编排 │ 监控告警      │
└────────────┬────────────────────────┘
             │ API
┌────────────▼────────────────────────┐
│      数据平面 (Device Agent)         │
│  设备驱动  │ 扫描执行 │ 本地存储      │
└─────────────────────────────────────┘
```

**优点**: 逻辑分离，可独立扩展，故障隔离  
**缺点**: 网络通信开销，部署复杂度提升  
**适用**: 中大型部署（50+设备）

#### 推荐方案: 方案A（单体）+ 未来可拆分
- **理由**: 初期专注核心功能，降低复杂度
- **演进路径**: 当设备数量>30时，拆分为方案B

### 2.2 编程语言与运行时评估

| 语言 | 跨平台 | 资源占用 | 网络库 | 扫描仪SDK | 安全性 | 推荐度 |
|------|--------|---------|--------|-----------|--------|--------|
| **Go** | ✅ 优秀 | ✅ 低 | ✅ 丰富 | ⚠️ 需绑定 | ✅ 静态编译 | ⭐⭐⭐⭐⭐ |
| **Python** | ✅ 好 | ⚠️ 中等 | ✅ 丰富 | ✅ 完善 | ⚠️ 依赖管理 | ⭐⭐⭐⭐ |
| **Rust** | ✅ 优秀 | ✅ 极低 | ✅ 丰富 | ⚠️ 生态较少 | ✅ 内存安全 | ⭐⭐⭐⭐ |
| **C#** | ⚠️ Windows优先 | ✅ 低 | ✅ 好 | ✅ 好 | ✅ 强类型 | ⭐⭐⭐ |
| **Java** | ✅ 好 | ❌ 高 | ✅ 丰富 | ✅ 完善 | ⚠️ JVM | ⭐⭐ |

#### 推荐: Go语言（首选）或Python（备选）

**选择Go的理由**:
- 静态编译为单一可执行文件，部署友好
- 原生跨平台（Windows/Linux）
- 协程模型适合高并发设备管理
- 内存占用低（<100MB可管理50+设备）
- 标准库强大，网络编程友好
- 无依赖运行时，避免安全软件误报

**选择Python的理由**:
- 开发速度快，原型验证友好
- SANE（Linux扫描库）绑定完善
- 丰富的图像处理库（Pillow、pdf2image）
- 如选Python，建议用PyInstaller打包为exe

### 2.3 扫描协议与设备兼容性

#### 主流扫描协议支持

| 协议 | 品牌覆盖 | 自动发现 | 复杂度 | 优先级 |
|------|---------|---------|--------|--------|
| **eSCL** (AirPrint Scan) | HP, Canon, Epson, Brother | ✅ mDNS | 低 | P0 |
| **WSD** (Web Services for Devices) | Windows-centric | ✅ WS-Discovery | 中 | P1 |
| **SANE** (Scanner Access Now Easy) | Linux通用 | ❌ 需配置 | 低 | P1 (Linux) |
| **TWAIN** | Windows传统 | ❌ 驱动依赖 | 高 | P2 |
| **ISIS** | 商用高速扫描仪 | ❌ 专有驱动 | 高 | P2 |

#### 设备发现机制

**mDNS/Bonjour发现（推荐）**:
- 使用库: `github.com/grandcat/zeroconf` (Go)
- 服务类型: `_scanner._tcp`, `_uscan._tcp`, `_ipp._tcp`
- 优点: 零配置，自动发现，跨平台

**手动添加**:
- IP地址 + 协议类型选择
- 自动协议探测（尝试eSCL→WSD→SANE）

### 2.4 存储与文件管理

#### 存储架构

```
┌─────────────────────────────────────────┐
│              存储分层设计                 │
├─────────────────────────────────────────┤
│  热存储 (SSD)  │  冷存储 (HDD/NAS)        │
│  - 当日扫描    │  - 历史归档              │
│  - 正在处理    │  - 备份副本              │
└─────────────────────────────────────────┘
```

#### 文件组织策略

**目录结构**:
```
<StorageRoot>/
├── devices/
│   └── <device_id>/          # 设备级目录
│       └── 2026/
│           └── 06-11/
│               ├── 10.30.25/     # 批次目录（扫描开始时间）
│               │   ├── page_001.jpg
│               │   └── metadata.json
│               └── 14.15.33/
├── temp/                     # 扫描临时文件
├── quarantine/              # 可疑文件隔离
└── logs/                    # 审计日志
```

#### 文件安全保障

| 安全措施 | 实现方式 | 目的 |
|---------|---------|------|
| 写时校验 | SHA256校验和 | 防止文件损坏 |
| 双副本 | 本地 + 网络存储 | 防丢失 |
| 原子写入 | 临时文件→重命名 | 防止不完整文件 |
| 定期校验 | 后台任务扫描 | 发现存储介质故障 |
| 加密存储 | AES-256（可选） | 敏感文件保护 |

#### 数据库选型（元数据管理）

| 方案 | 类型 | 优点 | 缺点 | 推荐 |
|------|------|------|------|------|
| **SQLite** | 嵌入式 | 零配置，单文件 | 并发一般 | ⭐⭐⭐⭐⭐ |
| **BoltDB** | KV嵌入式 | 纯Go，高性能 | 无SQL | ⭐⭐⭐⭐ |
| **PostgreSQL** | 关系型 | 功能强大 | 需独立部署 | ⭐⭐⭐ |

**推荐SQLite**: 单个文件存储，备份简单，事务支持ACID

### 2.5 安全与防护

#### 避免安全软件误报

| 风险点 | 防护措施 |
|--------|---------|
| 网络扫描 | 使用标准mDNS而非主动端口扫描 |
| 本地监听 | 仅绑定127.0.0.1，避免0.0.0.0 |
| 文件操作 | 限定在指定目录，避免系统目录 |
| 进程行为 | 单进程，无注入，无钩子 |
| 代码签名 | Windows使用有效证书签名 |
| 安装程序 | 使用标准安装包（MSI/Deb/RPM） |

#### 运行时安全

- **最小权限**: 不以管理员/root运行（除非必要）
- **资源限制**: cgroups/systemd限制CPU/内存
- **沙箱**: Linux可用seccomp，Windows可用AppContainer
- **输入验证**: 严格验证所有网络和设备输入

### 2.7 资源优化策略

#### 内存优化

| 策略 | 实现 | 效果 |
|------|------|------|
| 流式处理 | 边下载边处理，不缓存完整文件 | 降低峰值内存 |
| 对象池 | 重用HTTP client和buffer | 减少GC压力 |
| 分页加载 | 设备列表分页，不常驻内存 | 支持大量设备 |

#### CPU优化

- 扫描任务使用后台优先级
- 图片转换使用协程池限制并发数
- 避免忙等待，使用事件驱动

#### 网络优化

- HTTP keep-alive复用连接
- 设备状态缓存（5秒TTL）
- 后台增量同步而非轮询

## 三、技术栈建议

### 3.1 推荐技术栈（Go方案）

| 层级 | 组件 | 技术选型 | 理由 |
|------|------|---------|------|
| **语言** | 主要语言 | Go 1.22+ | 静态编译，高性能，跨平台 |
| **构建** | 构建工具 | Go Modules + Mage | 现代化构建，跨平台脚本 |

| | mDNS发现 | zeroconf | 成熟的Bonjour/Zeroconf实现 |
| | Web服务 | Gin或Echo | 轻量级，性能优秀 |
| **存储** | 元数据 | SQLite (mattn/go-sqlite3) | 嵌入式，单文件 |
| | 文件存储 | 本地文件系统 | 简单可靠 |
| | 缓存 | ristretto | 高性能内存缓存 |
| **图像** | 处理 | imaging + pdfcpu | 纯Go实现，无CGO |
| | PDF转换 | pdf2image（Python桥接） | 处理复杂PDF |
| **日志** | 结构化日志 | Zap | 高性能结构化日志 |
| | 审计 | 独立SQLite表 | 长期保留 |
| **配置** | 配置文件 | Viper | 支持多格式，热重载 |
| **监控** | 指标 | Prometheus client | 行业标准 |
| | 告警 | Alertmanager集成 | 外部告警 |
| **安全** | 加密 | 标准库crypto | 无需第三方 |
| | 校验 | Blake3 | 快速文件校验 |
